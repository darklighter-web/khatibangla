================================================================================
                           DEVELOPMENT GUIDELINES
================================================================================

1. MARIADB 10.11 STRICT MODE:
   - Aggregate aliases CANNOT be referenced in ORDER BY
   - MUST repeat full expressions: ORDER BY SUM(x) instead of ORDER BY alias
   - Use COALESCE() for null safety in SQL

2. PHP 8.0 SAFETY AUDIT (every file):
   - Use ?? operator for undefined array keys
   - Use intval() for numeric inputs
   - Use COALESCE() in SQL
   - Check for null dereference

3. FILE DELIVERY PROTOCOL:
   ✅ ALWAYS provide BOTH:
   - Git push to origin main
   - ZIP file structured for public_html/ extraction
   
   ✅ SQL migrations: SEPARATE files (never inside ZIP)
   
   ✅ Update this SYSTEM_INFO.txt for any feature changes

4. PAGE GUIDE SYSTEM:
   - Location: admin/includes/footer.php
   - JS object 'guides' keyed by page basename
   - ALWAYS update guides when modifying admin pages

5. ADMIN BUILDER FILE:
   - Correct: admin/pages/landing-page-builder.php
   - NOT: admin/pages/page-builder.php (different file)

6. FRAUDCHECKER API:
   - Full integration was built then ROLLED BACK

                          CHANGELOG — 2026-03-27
================================================================================

FIX #1 — ORDER STATUS NOT CHANGING
  Root Cause: The global window.confirm() override in admin/includes/header.php
  always returns `false` synchronously. Any code using `if(confirm(…)){action}`
  never executes the action block. This broke:
    • Inline row-menu status buttons (Confirm/Ship/Deliver/Cancel) in order-management.php
    • Bulk status change in order-management.php
    • Single-order courier uploads (Pathao/Steadfast/RedX) in order-view.php
    • Panel note clearing, SMS sending, lock force-clear in order-view.php
  Fix: Converted ALL `if(confirm(…))` and `if(!confirm(…))` patterns to use
  `await window._confirmAsync(…)` (the async Promise-based styled modal).
  Functions changed to `async function` where needed.
  Files changed: order-management.php, order-view.php, header.php

FIX #2 — COURIER UPLOAD RATE LIMITING + INTERACTIVE PROGRESS
  Old behavior: All selected orders uploaded in a single bulk POST request.
    No per-order progress, no rate limiting, no cooldown.
  New behavior:
    • Sequential upload: 1 order per second (1s minimum interval)
    • After 18 continuous uploads: 30-second cooldown with visible countdown
    • Per-order live progress: each row shows PENDING → UPLOADING → SUCCESS/ERROR
    • Consignment ID shown inline per order on success
    • Pause/Resume button during upload
    • ETA and elapsed time display
    • Cannot accidentally close modal during upload (prompts confirmation)
    • Animated progress bar with shimmer effect
    • Auto-scroll to currently uploading order
  File changed: order-management.php (openCourierUploadModal, _sequentialCourierUpload)

FIX #3 — CONSIGNMENT ID DISPLAY AFTER UPLOAD
  The courier_consignment_id was already stored in the `orders` table by both
  pathao-api.php and courier-bulk-upload.php. The order-management.php list
  already renders it in the "Upload" column when present. The fix ensures:
    • New sequential upload modal shows CID per-order in real time
    • "Refresh List" button reloads order table after upload completes
    • The order row in order-management.php displays CID with courier link
  Files: order-management.php (modal UI), courier-bulk-upload.php (already OK)

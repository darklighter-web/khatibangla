   - Aggregate aliases CANNOT be referenced in ORDER BY
   - MUST repeat full expressions: ORDER BY SUM(x) instead of ORDER BY alias

   - Use ?? operator for undefined array keys
   - Use intval() for numeric inputs
   - Use COALESCE() in SQL

  Files changed: order-management.php, order-view.php, header.php

CRITICAL: DATA INTEGRITY & CORE WORKFLOW FIXES

FIX #1 — LOCATION MAPPING BUG (Mirpur 14 → Mirpur 1)
  Root Cause: Zone scoring in auto-detect used `aliasAddr.includes(zn)` which
  matched "mirpur 1" as substring inside "mirpur 14". First-word matching with
  `includes(fw)` had same issue — "mirpur" matched all Mirpur zones equally.
  Fix: Replaced with word-boundary regex for exact zone name matching. Added
  number-aware scoring: zones with numbers (e.g. "Mirpur 14") now require the
  exact number to match via regex `mirpur\s*-?\s*14(?:\s|$|[^0-9])`. Wrong
  number matches get a -50 penalty. Exact bounded match gets +200 score.
  File: admin/pages/order-view.php (autoDetectLocation → scoredZones)

FIX #2 — ORDER CONFIRMATION VALIDATION
  Problem: Orders could be confirmed without customer name, phone, address, or
  products. No backend validation existed.
  Fix (Backend): Added strict validation in order-view.php confirm_order handler.
  Checks: name not empty, phone ≥10 digits, address not empty, ≥1 product,
  total > 0 (unless advance paid). Returns JSON error, iframe postMessage, or
  redirect with detail on failure.
  Fix (Frontend): Added validateConfirmOrder() onclick handler on confirm button.
  Checks same fields client-side and shows modal error before submit.
  Fix (Bulk): order-management.php bulk_status handler now validates required
  fields when confirming orders in bulk — skips with error message if missing.
  Files: admin/pages/order-view.php, admin/pages/order-management.php

FIX #3 — DISAPPEARING INCOMPLETE ORDERS (Navigation State Bug)
  Root Cause: Browser back/forward popstate handler only restored state from
  `e.state` object. If no state was stored (first page load), incomplete tab
  data was lost. Also courier filter bar incorrectly appeared on incomplete tab
  because 'incomplete' was missing from the no-courier-statuses list in JS.
  Fix: Added fallback URL parsing in popstate handler — if `e.state` is null,
  parses all params from `location.search` and re-fetches. Added 'incomplete'
  to `_noCourrierStatuses` array so courier bar stays hidden.
  File: admin/pages/order-management.php

OVERVIEW:
The product variation system now supports two modes:
1. LEGACY MODE (Single) - Original flat-row system where each row is one option
   (e.g. Color: Red, Size: L as separate rows with individual prices)
2. COMBINATION MODE - WooCommerce-style system where attributes are defined first,
   then all combinations are auto-generated (e.g. Frame Color × Lens Color × Size
   = Golden+Black+Large as one combination with its own price/stock/SKU/image)

NEW DATABASE TABLES:
- product_attributes: Stores attribute definitions per product
  Columns: id, product_id, attribute_name, attribute_values (JSON), is_visible,
           is_variation, sort_order
- product_variant_combinations: Stores each combination with its own pricing/stock
  Columns: id, product_id, combination_key (JSON), combination_label, sku,
           regular_price, sale_price, cost_price, stock_quantity, manage_stock,
           weight, variant_image, is_active, is_default, sort_order
- products.variation_mode: ENUM('legacy','combination') - which mode the product uses

BACKWARD COMPATIBILITY:
- Combination mode syncs to legacy product_variants table automatically
- Frontend product page works unchanged (reads from product_variants)
- order_items.variant_name format preserved
- Existing legacy products continue working without changes

STOCK TRACKING FIX:
- FIXED: Stock was previously deducted from BOTH products.stock_quantity AND
  product_variants.stock_quantity (double counting)
- NOW: For variable products, stock is deducted ONLY from product_variants.
  product.stock_quantity is then recalculated as SUM of all variant stocks.
- For simple products, only products.stock_quantity is adjusted (no change)

FILES CHANGED:
- admin/pages/product-form.php: Added combination mode UI (Attributes tab,
  Variations tab with Generate button + bulk actions, Addons tab), mode toggle,

HOW TO USE:

================================================================================
                  ORDER MANAGEMENT LOGIC UPDATES
================================================================================
Updated: 2026-04-03

── 1. SUPER ADMIN BYPASS ──

Super Admins (role_name = 'super_admin') now bypass ALL order state-change
restrictions. Detection via isSuperAdmin() in admin/includes/auth.php.

WHAT'S BYPASSED:
  • Edit immutability — Super Admin can edit orders in confirmed/shipped/delivered
    etc. states. Normal users see read-only mode for confirmed+ orders.
    File: order-view.php (save_order action, line ~67)
  • Status revert guard — Super Admin can revert confirmed→processing, 
    shipped→pending, etc. Normal users are blocked from reverting post-confirm
    orders to pre-confirm statuses.
    Files: order-view.php (update_status action), order-management.php (bulk_status)
  • Frontend dropdown — Super Admin sees ALL 15 statuses in the status dropdown
    regardless of current status. Normal users see filtered transitions.
    File: order-view.php (statusSelect dropdown)
  • Read-only mode — Super Admin sees the full editable form, not the read-only
    locked version shown to other roles for confirmed+ orders.
    File: order-view.php ($_isShippedLocked flag)
  • Row menu actions — Super Admin sees Edit/Confirm/Ship/Deliver/Cancel buttons
    in the row context menu (⋮) regardless of order status.
    File: order-management.php (toggleRowMenu function)
  • A red "Super Admin" badge appears next to "Order Actions" in order-view.php
    to indicate elevated permissions are in effect.

── 2. RETURN-TO-SOURCE NAVIGATION ──

When closing an order view, users return to the specific sub-panel they came
from (e.g., Processing, Shipped, Confirmed) instead of the default All Orders.

MECHANISM:
  • A `return_status` query parameter is passed through all order navigation paths
  • order-view.php reads this param (or extracts from HTTP_REFERER as fallback)
    and builds $_returnQs = "status=<value>&" for all redirect URLs

WHERE RETURN_STATUS IS INJECTED:
  • Order row template links: invoice number link + Open button
    File: order-row-tpl.php (both pre-conf direct link and post-conf modal link)
  • Modal iframe: openOrderModal() reads status from current URL params
    File: order-management.php (openOrderModal function)
  • Row context menu: Open link includes return_status
    File: order-management.php (toggleRowMenu function)

WHERE RETURN_STATUS IS CONSUMED (redirects):
  • After save_order / confirm_order → order-management.php?status=X
  • After update_status → order-management.php?status=X&highlight=ID
  • After mark_fake → order-management.php?status=X&highlight=ID
  • After add_note (non-AJAX fallback) → order-management.php?status=X
  • After clear_panel_notes → order-management.php?status=X
  • "Back to List" link in order actions panel → order-management.php?status=X
  • "Back to Orders" link on lock screen → order-management.php?status=X
  • Modal close: OM.refresh() already preserves current tab (AJAX state)

FILES CHANGED:
  order-view.php, order-management.php, order-row-tpl.php


================================================================================
              SECTION VISIBILITY (PER-DEVICE TOGGLES)
================================================================================
Updated: 2026-04-04

LOCATION: Admin → Settings → Header & Nav tab (top panel)

Each header/nav section can be independently shown or hidden for Desktop (PC)
and Mobile devices using toggle switches.

SECTIONS CONTROLLED:
  1. Top Bar / Announcement   — vis_topbar_desktop, vis_topbar_mobile
  2. Main Header              — vis_main_header_desktop, vis_main_header_mobile
  3. Category Navigation Bar  — vis_cat_nav_desktop, vis_cat_nav_mobile
  4. Mobile Bottom Nav        — vis_mobile_bottom_nav_desktop, vis_mobile_bottom_nav_mobile
  5. Mobile Sticky Buy Bar    — vis_mobile_sticky_bar_desktop, vis_mobile_sticky_bar_mobile

HOW IT WORKS:
  • Settings stored in site_settings table as vis_{section}_{device} = '0' or '1'
  • Default: all sections visible ('1') on all devices
  • CSS media queries injected in includes/header.php <style> block
  • Desktop = min-width: 1024px, Mobile = max-width: 1023px
  • Uses !important to override existing Tailwind classes
  • CSS classes on elements: vis-topbar, vis-main-header, vis-cat-nav,
    vis-mobile-bottom-nav, vis-mobile-sticky-bar

ADMIN UI:
  • Indigo (purple-ish) toggle = Desktop, Purple toggle = Mobile
  • Hidden input value="0" + checkbox value="1" pattern ensures proper save
  • Checkbox map in settings.php save handler covers all vis_* keys

FILES:
  • Admin UI:    admin/pages/settings.php (Header & Nav tab, checkbox map)
  • Frontend:    includes/header.php (CSS + class on topbar, header, cat nav)
  • Footer:      includes/footer.php (class on mobile bottom nav)
  • Product:     pages/product.php (class on mobile sticky buy bar)
